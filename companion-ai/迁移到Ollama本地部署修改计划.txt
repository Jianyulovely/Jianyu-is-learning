  Companion-AI 项目迁移到 Ollama 本地部署 - 修改计划
═══════════════════════════════════════════════════════════════════════════════

📅 创建日期：2026-04-02
🎯 目标：将 companion-ai 项目从直接加载模型改为使用本地 Ollama 部署
📚 参考：telegram_bot/ 项目中的 Ollama 封装实现

───────────────────────────────────────────────────────────────────────────────
一、当前架构分析
───────────────────────────────────────────────────────────────────────────────

【现状】
1. llm/api.py：直接使用 transformers 加载模型到 GPU，提供 /chat 接口
2. bot/handler.py：通过 httpx 调用本地 LLM 服务的 /chat 接口
3. 使用了 SSH 隧道相关配置（IPv4 强制绑定、Connection: close）
4. 模型加载在 llm/model_loader.py 中实现

【问题】
- 需要手动管理 GPU 内存和模型加载
- SSH 隧道增加了部署复杂度
- 不够灵活，切换模型需要重启服务

───────────────────────────────────────────────────────────────────────────────
二、目标架构
───────────────────────────────────────────────────────────────────────────────

【新架构】
1. 使用本地 Ollama 服务管理模型
2. llm/api.py 改为 FastAPI 封装层，转发请求到 Ollama
3. 使用 Ollama 的 /api/generate 端点（因为需要自己拼接提示词）
4. 添加 Bearer Token 认证保护 API
5. 移除所有 SSH 隧道相关代码

【优势】
- Ollama 自动管理模型加载和 GPU 资源
- 部署更简单，只需本地运行 Ollama
- 可以通过 Ollama 轻松切换模型
- API 层更轻量，只负责转发和认证

【步骤 3】重写 LLM API 服务（llm/api.py）
────────────────────────────────────────
修改文件：companion-ai/llm/api.py

参考：telegram_bot/api.py 的实现模式

核心改动：
1. 移除所有模型加载相关代码（transformers, torch）
2. 改为 FastAPI 封装 Ollama 的 /api/generate 接口
3. 支持非流式响应（companion-ai 当前不需要流式）
4. 本地服务无需认证保护

新的 API 结构：
```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import httpx
import uvicorn

# 请求模型
class GenerateRequest(BaseModel):
    prompt: str              # 完整的拼接好的提示词
    max_new_tokens: int = 200
    temperature: float = 0.85
    top_p: float = 0.9
    repetition_penalty: float = 1.1

# 响应模型
class GenerateResponse(BaseModel):
    reply: str
    usage: dict

# /generate 端点
@app.post("/generate")
async def generate(req: GenerateRequest):
    # 转发到 Ollama
    payload = {
        "model": config.OLLAMA_MODEL,
        "prompt": req.prompt,
        "stream": False,
        "options": {
            "num_predict": req.max_new_tokens,
            "temperature": req.temperature,
            "top_p": req.top_p,
            "repeat_penalty": req.repetition_penalty,
        }
    }
    async with httpx.AsyncClient(timeout=120) as client:
        r = await client.post(config.OLLAMA_GEN_URL, json=payload)
        r.raise_for_status()
        data = r.json()
        return GenerateResponse(
            reply=data["response"],
            usage={"input_tokens": 0, "output_tokens": 0}  # Ollama 不返回详细 token 数
        )
```

关键点：
- 使用 /api/generate 而不是 /api/chat（因为需要自己拼接提示词）
- 保持与原 API 相同的请求/响应格式，减少 handler.py 的改动
- 本地服务无需认证，简化代码


【步骤 4】修改 Bot Handler 的 LLM 调用逻辑（bot/handler.py）
────────────────────────────────────────
修改文件：companion-ai/bot/handler.py

需要修改的函数：_call_llm

原实现：
- 接收 system_prompt 和 history
- 组装成 messages 列表
- 调用 /chat 接口

新实现：
- 接收 system_prompt 和 history
- 手动拼接成完整的 prompt 字符串
- 调用 /generate 接口
- 添加 Bearer Token 认证
- 移除 SSH 隧道相关的配置（local_address, Connection: close）

修改后的代码：
```python
async def _call_llm(system_prompt: str, history: list[dict]) -> str:
    # 手动拼接提示词
    prompt_parts = [f"System: {system_prompt}\n"]
    for msg in history:
        role = msg["role"]
        content = msg["content"]
        if role == "user":
            prompt_parts.append(f"User: {content}\n")
        elif role == "assistant":
            prompt_parts.append(f"Assistant: {content}\n")
    prompt_parts.append("Assistant: ")
    full_prompt = "".join(prompt_parts)

    payload = {
        "prompt": full_prompt,
        "max_new_tokens": 200,
        "temperature": 0.85,
        "top_p": 0.9,
        "repetition_penalty": 1.1,
    }

    async with httpx.AsyncClient(timeout=120.0) as client:
        resp = await client.post(
            f"{config.LLM_API_URL}/generate",
            json=payload,
        )
        resp.raise_for_status()
        return resp.json()["reply"]
```

移除的代码：
- `transport = httpx.AsyncHTTPTransport(local_address="0.0.0.0")`
- `headers={"Connection": "close"}`
- `headers={"Authorization": f"Bearer {config.LLM_API_TOKEN}"}`
- 相关的 SSH 隧道注释

───────────────────────────────────────────────────────────────────────────────
六、重要注意事项
───────────────────────────────────────────────────────────────────────────────

【1. 提示词拼接格式】
- 当前使用简单的 "System: ... User: ... Assistant: ..." 格式
- 如果效果不好，可能需要调整为模型特定的格式
- Qwen 模型可能需要特殊的 token 格式，需要测试验证

【2. Ollama 模型管理】
- 使用 `ollama list` 查看已下载的模型
- 使用 `ollama pull <model>` 下载新模型
- 使用 `ollama rm <model>` 删除模型

【3. 性能考虑】
- Ollama 会自动管理 GPU 内存
- 如果需要指定 GPU，在启动 Ollama 时设置 CUDA_VISIBLE_DEVICES
- 小模型（0.5B）响应速度快，但质量可能不如大模型

【4. 错误处理】
- 如果 Ollama 服务未启动，API 层会返回 500 错误
- 建议在 API 层添加更详细的错误信息
- Bot 层已有错误处理，会返回友好的错误提示

【5. 日志监控】
- 建议在 llm/api.py 中添加详细日志
- 记录每次请求的 prompt 长度、响应时间等
- 便于后续优化和问题排查

十、总结
───────────────────────────────────────────────────────────────────────────────

【核心改动】
本次迁移的核心是将 companion-ai 从"直接加载模型"改为"通过 Ollama 管理模型"，
主要优势包括：
1. 简化部署：不再需要 SSH 隧道，本地运行即可
2. 降低复杂度：移除 torch/transformers 依赖，API 层更轻量
3. 提高灵活性：通过 Ollama 可以轻松切换不同模型
4. 更好的资源管理：Ollama 自动管理 GPU 内存

【关键技术点】
1. 使用 Ollama 的 /api/generate 端点（而非 /api/chat）
   - 原因：companion-ai 需要通过 PromptEngine 自己拼接提示词
   - 需要手动将 messages 列表转换为单个 prompt 字符串

2. 保持 API 接口兼容性
   - 新的 llm/api.py 保持与原来相同的请求/响应格式
   - 减少对 bot/handler.py 的改动

3. 简化架构
   - 本地服务无需认证保护
   - 减少配置项和代码复杂度

【后续优化建议】
1. 添加流式响应支持（如果需要更快的用户体验）
2. 优化提示词拼接格式，适配不同模型的特殊格式
3. 添加请求缓存机制，减少重复请求
4. 实现更完善的错误处理和重试机制
5. 添加性能监控和日志分析
